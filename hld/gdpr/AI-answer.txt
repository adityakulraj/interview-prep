High-Level Design
GDPR-Compliant Cross-Region Request Sharing (EU → NA)
1. Overview

We need a system where:

A user in the EU region submits a form/request.

The request is processed and stored in the EU data center only.

Users/clients in the NA region can view and act on the request.

PII or GDPR-protected data must never be stored in the NA data center.

The system should support asynchronous request delivery, high consistency, fault tolerance, and reliable storage.

This design ensures GDPR compliance by keeping all personal data in the EU while still enabling NA users to collaborate on sanitized request content.

2. Problem Statement

Build a multi-region system that allows:

EU users to create requirement documents/forms.

Documents to be stored as JSON in the EU region.

NA clients/users to access the document in a sanitized/redacted form.

NA users to comply with the request and provide feedback.

Feedback from NA to be sent back to EU without violating GDPR.

3. Key Assumptions

To make the design concrete, I’m assuming:

EU users are the data owners and may submit forms containing PII.

NA users need business context, but not raw personal data.

A redaction/sanitization layer can derive a GDPR-safe representation of the document.

Only sanitized payloads are allowed outside the EU region.

The system is write-authoritative in EU for user data and original requests.

Feedback from NA is generally non-PII, but will still be validated before syncing back to EU.

4. Functional Requirements
Core Functional Requirements

EU users submit requirement documents through a form.

Requirement documents are stored as JSON.

Original documents containing PII must remain in the EU data center only.

NA clients can view the document only after PII removal/redaction.

NA clients can act on the request.

NA clients can submit feedback/suggestions.

Feedback is asynchronously sent back to the EU region.

No raw EU personal data may be stored in the NA region.

5. Non-Functional Requirements

GDPR compliance: all EU user personal data remains in EU.

Reliability: EU user data must be stored reliably.

Fault tolerance: component failure should not lose requests.

High consistency for original request state and compliance status.

Asynchronous communication between regions.

Auditability of access, redaction, sync, and feedback flows.

Scalability to support large numbers of requests across regions.

No raw PII, no original document

7. High-Level Flow
7.1 Request Creation Flow

EU user submits form.

Request lands in EU API Gateway.

Request Service validates and stores original JSON in EU DB / document store.

PII Redaction Service generates a sanitized version:

remove name, email, phone, address, user identifiers

mask sensitive free-text fields if needed

EU publishes an event:

RequestCreatedSanitized

Event is asynchronously replicated/sent to NA.

NA consumer receives only sanitized payload.

Sanitized request is stored in NA read model store.

NA users can now view and process the request.

7.2 Compliance / Action Flow in NA

NA client opens sanitized request.

NA service retrieves sanitized document from NA read store.

Client processes request and marks action/compliance state.

Feedback is added.

NA publishes:

RequestProcessed

FeedbackSubmitted

Event is sent back to EU asynchronously.

EU Feedback Ingestion Service updates authoritative request state.

7.3 Feedback Suggestion Flow

After complying with the request, NA client submits feedback.

Feedback is validated for accidental PII leakage.

Safe feedback is sent to EU asynchronously.

EU persists feedback with the original request.

EU user can view final status + feedback.

12. Consistency Model
Strong consistency

Needed in the EU region for:

original request creation

request status

feedback persistence

audit records

Eventual consistency

Acceptable for:

propagation of sanitized documents to NA

feedback propagation from NA to EU

This satisfies:

high consistency for source-of-truth data

asynchronous send of requests between regions

13. GDPR Compliance Controls
Mandatory controls

Data localization: raw personal data stays only in EU.

Sanitization before transfer: only redacted JSON moves to NA.

No NA persistence of PII.

Audit logs for every access and transfer.

Encryption at rest and in transit.

Data retention policy by region.

Access control / RBAC:

EU admin: full view

NA client: sanitized view only

DLP / validation on outgoing feedback to prevent reintroduction of PII.

14. Reliability and Fault Tolerance
Reliability

EU DB configured with replication inside EU region.

Durable message queue for cross-region events.

Idempotent consumers in NA and EU.

Retry with dead-letter queue for failed sync.

Fault tolerance

If NA is down:

EU continues accepting requests.

Sanitized events remain queued until NA recovers.

If EU feedback ingestion is down:

NA feedback events remain in queue and retry later.

If redaction service fails:

request remains stored in EU

request is not released to NA until sanitization succeeds

15. Security Design

TLS for all service-to-service traffic

KMS-managed encryption keys

Role-based access control

Token-based authentication (OAuth/JWT)

Request signing for inter-region messages

Immutable audit log for compliance review

PII classification service before export

16. Failure Scenarios
Case 1: Redaction failure

Original request stored in EU

Mark request as PENDING_SANITIZATION

Do not sync to NA

Case 2: Queue delivery failure

Retry with exponential backoff

Dead-letter queue for manual review

Case 3: NA feedback contains accidental PII

DLP validation blocks submission

Return validation error

Ask NA client to revise feedback

Case 4: Duplicate events

Use eventId + version for idempotency

Ignore already processed events

17. Scalability Considerations

Stateless API services behind load balancers

Partition requests by requestId / region

Separate read store in NA for fast access

Async queues absorb traffic spikes

Redaction service can scale independently

18. Open Design Decisions

These should be clarified during detailed design:

What exact fields count as PII?

Can free-text fields contain hidden PII?

Should NA see masked values or fully removed fields?

Is feedback always non-PII, or must it also be sanitized?

What SLA is required for EU → NA sync?

Is 100% reliability truly required, or practically “no data loss” with durable storage?

19. Recommended Tech Stack (Example)

This is optional, but a typical implementation could be:

API layer: Java / Go / Node.js

DB in EU: PostgreSQL / CockroachDB / Aurora (EU only)

Document store: S3-compatible EU-only object store

Messaging: Kafka / SNS+SQS / PubSub equivalent

Redaction: DLP/PII detection service + custom rules

Cache/read model in NA: sanitized PostgreSQL / DynamoDB / Redis

Observability: Prometheus, Grafana, ELK

20. Final HLD Summary

The proposed design uses an EU-authoritative architecture with sanitized asynchronous replication to NA.

EU is the source of truth

PII never leaves EU

NA only sees redacted data

Cross-region sync is async and event-driven

Feedback flows back to EU

Strong consistency is maintained in EU

Fault tolerance and auditability are built in

This gives a GDPR-compliant design that supports cross-region collaboration without violating data residency requirements.
